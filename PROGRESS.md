## Sprint 3 — POST /upload (Full Ingestion Pipeline)

### What was built
- `backend/ingestion/router.py` — full pipeline endpoint:
  1. File type validation (PDF / DOCX only)
  2. Save raw file to `data/raw_uploads/<uid>_<filename>`
  3. `asyncio.to_thread(extract_raw_text)` — pymupdf for PDF, pandoc for DOCX
  4. `asyncio.to_thread(preprocess_syllabus)` — Claude Haiku with SYLLABUS_TEMPLATE.md prompt
  5. Save processed markdown to `data/processed_markdown/<uid>_<stem>.md`
  6. Extract `course_code` and `course_title` via regex on `**Course Code & Title**` line
  7. ORM insert Syllabus row + `db.flush()` to get id
  8. `chunk_markdown()` → filter empty sections
  9. Single batched `embed_texts()` call (all chunks in one OpenAI request)
  10. Bulk chunk insert via `db.execute(text(...), params_list)` with `CAST(:embedding AS vector)`
  11. `db.commit()` — single transaction for syllabus + all chunks
- `main.py` updated to register ingestion router and configure logging

### Errors & Fixes

**1. Wrong model ID**
- Error: `404 — model: claude-haiku-4-5-20241022`
- Fix: Correct ID is `claude-haiku-4-5-20251001`. Updated `.env`, `config.py`, `.env.example`

**2. Empty chunk content crashing OpenAI embedding API**
- Error: `Invalid 'input[0]': input cannot be an empty string`
- Cause: Claude output starts with a `# Title` heading with no body; chunker produced an empty-content chunk
- Fix: Filter `chunks = [c for c in raw_chunks if c["content"].strip()]` before embedding

**3. pgvector + asyncpg codec conflict (the hard one)**

Three failed approaches before the working fix:

| Attempt | Error | Why it failed |
|---|---|---|
| `connect_args={"init": async_fn}` | `TypeError: connect() got unexpected keyword argument 'init'` | `init` is a pool argument, not a `connect()` argument |
| `@event.listens_for(engine.sync_engine, "connect")` with `dbapi_connection.run_sync(fn)` | `AttributeError: 'AsyncAdapt_asyncpg_connection' has no attribute 'run_sync'` | Method is `run_async`, not `run_sync` |
| `@event.listens_for` with `dbapi_connection.run_async(pgvector.asyncpg.register_vector)` | `ValueError: could not convert string to float: '[0.1, ...]'` | `pgvector.sqlalchemy.Vector.bind_processor` already converts list → string; registering the asyncpg binary codec causes double-encoding (codec receives a string, expects list/array) |

- **Root cause**: `pgvector.sqlalchemy` and `pgvector.asyncpg` fight over encoding. The SQLAlchemy type's `bind_processor` converts Python list → `'[x, y, ...]'` string. If the asyncpg binary codec is also registered, it receives the string and tries to binary-encode it as a vector, which fails.
- **Fix**: Don't register the asyncpg codec at all. Use `CAST(:embedding AS vector)` in all vector INSERT/SELECT SQL. This forces the parameter to be treated as `text` by asyncpg, and PostgreSQL applies the cast. Works reliably with text protocol. Applied in `vector_store.insert_chunk`, `vector_store.vector_search`, and `ingestion/router.py` bulk insert.

### Retrieval Architecture Rework (multi-course grading query fix)

**Root problem:** When multiple syllabi were selected, grading breakdowns for some courses (CSE 563, CSE 564) were missing from LLM responses.

**Cause 1 — Chunker cross-section overlap bug:**
The chunker was prepending the previous section's tail into the next section's body. "Grading Policies" (thin section, ~25 chars of real content) was receiving course-objectives text from the preceding "Expected Learning Outcomes" section. This corrupted both the chunk content and its embedding.
- Fix: Removed overlap entirely. Each section is its own clean chunk.
- 464 existing chunks updated in DB by re-running chunker on stored `processed_markdown`
- All 649 chunks re-embedded in batches of 100 via OpenAI API

**Cause 2 — Cross-encoder scoring bias:**
`ms-marco-MiniLM-L6-v2` is trained on web passages; it scores structured markdown tables (grade breakdowns) poorly relative to prose. When reranking per-syllabus candidates globally, grading tables consistently lost to course-info/instructor chunks.
- Fix: Removed cross-encoder from the per-syllabus retrieval path. Hybrid search RRF score (vector cosine + FTS) is a reliable enough relevance signal within a single syllabus.

**Cause 3 — Short grading chunks losing in vector search:**
CSE 565's Grade Breakdown chunk (201 chars) was ranked below "Additional Course Information" (4534 chars, many keywords) in vector + RRF scoring even for grading-specific queries.
- Fix: Guaranteed grade chunk inclusion via direct header-name lookup:
  ```sql
  WHERE c.syllabus_id = :sid
    AND c.section_header ILIKE ANY(ARRAY[
          '%grade breakdown%', '%grade scale%',
          '%grading%', '%exam mode%'
        ])
  ```
  These chunks are always appended to the context regardless of retrieval score.

**Final per-syllabus retrieval path:**
1. Hybrid search top-3 RRF chunks per selected syllabus
2. Guaranteed fetch of grading/exam chunks by section header ILIKE
3. Dedup by chunk id (seen set)
4. No cross-encoder reranking

**Final global path (no syllabus filter):**
- Retrieval k = max(15, n_syllabi × 4)
- Rerank k = max(5, n_syllabi × 2)
- Cross-encoder reranking retained (global path query patterns are more open-ended; prose-heavy results are fine)

## Key Technical Decisions & Gotchas

| Decision | Rationale |
|---|---|
| `CAST(:embedding AS vector)` in all vector SQL | Avoids asyncpg binary codec conflict with pgvector.sqlalchemy's bind_processor |
| `asyncio.to_thread` for file parsing and LLM preprocessing | Both are sync/blocking; running in thread prevents event loop stalls |
| Single bulk `embed_texts()` call per upload | Cheaper and faster than one API call per chunk |
| Single transaction for syllabus + all chunks | `db.flush()` to get syllabus.id, then bulk insert chunks, then `db.commit()` |
| `content_tsv` column + trigger | Pre-built tsvector avoids `to_tsvector()` recomputation on every FTS query; GIN index only works on a stored column |
| Lazy CrossEncoder load in `_get_reranker()` | Avoids slow startup and TF/Keras import errors at server boot |
| Model IDs in `.env` | `CHAT_MODEL` and `PREPROCESSING_MODEL` — swap to sonnet without code changes |
